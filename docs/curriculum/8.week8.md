# Week 8 — Mini Capstone

## Theme
**Mini Capstone — End-to-End ML Workflow** 🧪

## Goals (By End of Week 8)
- We complete an **end-to-end ML workflow** that is *reproducible, leakage-safe, and defensible*.
- We perform **evidence-based model selection** using **CV mean + variance** (not a lucky split).
- We evaluate once on a **true holdout test set** and report results honestly.
- We ship a concise **Model Defense** report that explains *what we did and why it is valid*.

---

## Structure (Capstone Phases)

### Phase 0 — Project Setup & Rules
**Focus:** Reproducibility + evaluation discipline

- We define the scope:
  - Task type: **Classification** (recommended for Titanic-like data) or **Regression** (house prices-style data).
  - **Primary metric** (selected before tuning).
  - **Secondary metrics** (context only; not used to pick the winner).
- We lock evaluation rules:
  - **Test set is sacred** → we touch it once at the end.
  - All preprocessing happens **inside a Pipeline**.
  - We compare models using **the same CV protocol**.
  - We fix random seeds and record them.

✅ **Outcome:** We have a project skeleton and evaluation rules that prevent self-deception.

**Checkpoint Questions**
- What exactly makes a test set “sacred”?
- Which step most commonly causes leakage in beginner ML projects?

---

### Phase 1 — Data & Split Design
**Focus:** Clean splits + baseline-ready data

- We establish a split design (typical pattern):
  - `train` (for fitting + CV)
  - `test` (final evaluation only)
- We verify leakage risks:
  - Any “future-looking” fields are excluded (target leakage).
  - We do not compute global scalers/encoders before splitting.
- We define the feature schema:
  - Numeric vs categorical columns (for mixed tabular data).
  - Missing value policy (impute strategy).

✅ **Outcome:** We have a leakage-safe dataset setup and split strategy.

**Checkpoint Questions**
- Why is scaling the full dataset before CV invalid?
- What is target leakage in one sentence?

---

### Phase 2 — Baseline Model
**Focus:** A simple model we can defend

- We build a baseline Pipeline:
  - Preprocessing (impute/encode/scale as needed)
  - Baseline model (e.g., Logistic Regression for classification, Ridge for regression)
- We run CV and capture:
  - Mean score
  - Standard deviation across folds
  - Fold scores (distribution)

✅ **Outcome:** We have a baseline score distribution and a stable reference point.

**Checkpoint Questions**
- Why do we prefer a CV distribution over a single split result?
- What does “high variance across folds” signal?

---

### Phase 3 — Candidate Models
**Focus:** Comparing alternatives under the same protocol

- We select 2 candidate models (typical examples):
  - Classification: RandomForest / Gradient Boosting (optional)
  - Regression: RandomForestRegressor / GradientBoostingRegressor
- We enforce fairness:
  - Same preprocessing strategy (Pipeline)
  - Same CV splitter (StratifiedKFold for classification, KFold for regression)
  - Same metric for selection

✅ **Outcome:** We have a fair comparison across models using identical evaluation constraints.

**Checkpoint Questions**
- Why is it invalid to let one model see different preprocessing than another?
- When might we prefer a slightly lower mean but much lower variance?

---

### Phase 4 — Hyperparameter Tuning
**Focus:** CV-based tuning with bounded search

- We tune hyperparameters using:
  - `GridSearchCV` for small, curated grids
  - `RandomizedSearchCV` for wider search efficiently
- We keep tuning controlled:
  - We tune a few meaningful knobs (not everything).
  - We track top configurations and their CV distributions.
- We select a winner using:
  - CV mean + variance
  - Simplicity and stability
  - Metric alignment to the problem (imbalance-aware if needed)

✅ **Outcome:** We have a tuned best candidate chosen using CV evidence.

**Checkpoint Questions**
- Why does “best CV score” still not equal “true generalization”?
- When is RandomizedSearchCV preferred over GridSearchCV?

---

### Phase 5 — Final Holdout Test
**Focus:** Honest final reporting

- We refit the selected best pipeline on full training data.
- We evaluate once on the holdout test set and report:
  - Primary metric (the one we committed to early)
  - Secondary metrics for context
  - Confusion matrix / error breakdown (optional but useful)

✅ **Outcome:** We have a final, honest test score and supporting context.

**Checkpoint Questions**
- Why is repeated test evaluation a form of leakage?
- What is the difference between model selection and model assessment?

---

### Phase 6 — Model Defense Report
**Focus:** A short document we can stand behind

We produce a 1–2 page report covering:
- **Problem framing**
  - Task type, target definition, dataset summary
- **Metric choice**
  - Primary metric and why it matches the goal
- **Validation design**
  - Split strategy, CV strategy, leakage prevention steps
- **Results**
  - Baseline vs tuned model comparison (CV mean + std)
  - Final holdout test performance (single evaluation)
- **Risks and next steps**
  - Known limitations (data, features, imbalance, drift)
  - Improvement plan (data/features, calibration, thresholding, monitoring)

✅ **Outcome:** We can justify the workflow and results without hand-waving.

**Checkpoint Questions**
- If someone doubts the result, which evidence do we show first?
- What are the top 2 risks we should monitor after deployment?

## Reference Code Skeletons (Minimal, Clean)

### Skeleton 1 — CV Evaluation (Distribution)

```python
import numpy as np
from sklearn.model_selection import cross_val_score

def cv_summary(estimator, X, y, cv, scoring):
    scores = cross_val_score(estimator, X, y, cv=cv, scoring=scoring)
    return {
        "mean": float(np.mean(scores)),
        "std": float(np.std(scores)),
        "folds": scores,
    }
```

### Skeleton 2 — GridSearch with Pipeline (Leakage-Safe)

```python
from sklearn.model_selection import GridSearchCV

def tune_with_cv(pipe, param_grid, cv, scoring, n_jobs=-1):
    gs = GridSearchCV(
        estimator=pipe,
        param_grid=param_grid,
        cv=cv,
        scoring=scoring,
        n_jobs=n_jobs,
    )
    return gs
```

### Skeleton 3 — Final Test Evaluation (Single Use)

```python
def final_test_evaluate(best_model, X_train, y_train, X_test, y_test, scorer_fn):
    best_model.fit(X_train, y_train)
    return scorer_fn(best_model, X_test, y_test)
```
