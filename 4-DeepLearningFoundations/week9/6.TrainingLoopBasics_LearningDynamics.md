### ***2) What Is a Training Loop?***

A training loop is the repeated process used to train a neural network on data.

At a high level:

```text
1) take a batch of data
2) run a forward pass
3) compute loss
4) run backpropagation
5) update parameters
6) repeat
```

That happens again and again across the dataset.

Then after going through the whole dataset once, we call that one **epoch**.

So neural network training is not one giant update.
It is many small updates repeated over time.

---

### ***3) Epoch, Batch, and Mini-Batch***

These terms are easy to mix up, so we should lock them down clearly.

#### Epoch
An **epoch** means one full pass through the training dataset.

If we have 1,000 training examples and the model has seen all 1,000 once, that is one epoch.

#### Batch
A **batch** is a chunk of training examples processed together in one step.

If batch size = 100, then:
- 100 examples are used in one forward/backward/update cycle

#### Mini-Batch
A **mini-batch** is just a batch smaller than the full dataset.

In practice, most deep learning training uses mini-batches.

So if:
- dataset size = 1,000
- batch size = 100

then:
- 10 batches make 1 epoch

That means:
- 1 epoch = 10 update steps

#### Useful picture

```text
dataset → split into batches → each batch causes one update
all batches completed once → 1 epoch
```

---

### ***4) Why Not Use the Whole Dataset at Once?***

In theory, we could compute loss and gradients using the whole dataset at once.

But mini-batches are usually better in practice because:
- they are more memory efficient
- they are computationally manageable
- they often help optimization progress more smoothly
- they allow frequent parameter updates

So instead of waiting for the full dataset, the model learns in smaller chunks.

That is why batch-based training is standard.

---

### ***5) One Training Iteration: The Core Sequence***

One training iteration on one batch usually looks like this:

```text
1) zero old gradients
2) do forward pass
3) compute loss
4) do backward pass
5) update parameters
```

In PyTorch-like language, this often becomes:

```python
optimizer.zero_grad()
y_pred = model(X_batch)
loss = loss_fn(y_pred, y_batch)
loss.backward()
optimizer.step()
```

This is the core deep learning ritual.

Let’s understand each piece.

---

### ***6) `optimizer.zero_grad()`***

Gradients in frameworks like PyTorch are accumulated by default.

That means if we do not clear them, new gradients can pile on top of old ones.

So before each training step, we usually do:

```python
optimizer.zero_grad()
```

This means:
- clear out gradients from the previous step
- start fresh for the current batch

Without this, gradient values can become incorrect for the current step.

So this line is housekeeping, but very important housekeeping.

---

### ***7) Forward Pass***

Next:

```python
y_pred = model(X_batch)
```

This is the **forward propagation** step.

The batch of inputs moves through the network:
- weighted sums
- activations
- final output

The result is the model’s prediction for that batch.

So this step answers:

**Given the current weights and biases, what does the model predict?**

---

### ***8) Loss Computation***

Next:

```python
loss = loss_fn(y_pred, y_batch)
```

This compares:
- predicted output
- true target

and returns a scalar error value.

This tells us how wrong the model is on the current batch.

That loss is the quantity the model will try to reduce.

So now the question becomes:

**How should the parameters change so this loss gets smaller next time?**

That is where the backward pass comes in.

---

### ***9) Backward Pass***

Next:

```python
loss.backward()
```

This triggers backpropagation.

It computes gradients for all the parameters in the network.

So after this line runs, each weight and bias has gradient information attached to it.

That gradient tells us:
- direction of useful change
- magnitude of useful change

This is where the model stops merely observing error and starts preparing to learn from it.

---

### ***10) Optimizer Step***

Next:

```python
optimizer.step()
```

This updates the parameters using the gradients.

If we are using gradient descent or a related optimizer, this step typically does something like:

```text
new parameter = old parameter - learning_rate * gradient
```

So this is the actual learning move.

Without `step()`, gradients may exist, but the parameters do not change.

That means:
- `backward()` computes learning signals
- `step()` applies them

---

### ***11) Put It All Together***

One batch training step is:

```python
optimizer.zero_grad()
y_pred = model(X_batch)
loss = loss_fn(y_pred, y_batch)
loss.backward()
optimizer.step()
```

That is the skeleton.

Every larger training script is basically this skeleton wrapped in loops, monitoring, logging, validation, and extra features.

So if this part is clear, the rest is mostly structure around it.

---

### ***12) Training Loss vs Validation Loss***

Now we need to understand two very important curves.

#### Training Loss
Training loss is the loss computed on batches from the training data.

This tells us:
- how well the model is fitting the data it is learning from

#### Validation Loss
Validation loss is computed on a separate validation set that is not used for parameter updates.

This tells us:
- how well the model generalizes beyond the training batches

So:
- **training loss** = fit on seen training data
- **validation loss** = generalization check on held-out validation data

This distinction matters a lot.

A model can get better on training loss while getting worse on validation loss.

That is the beginning of overfitting.

---

### ***13) Underfitting, Good Fit, and Overfitting***

#### Underfitting
Underfitting means the model is too weak or not trained enough.

Signs:
- training loss is still high
- validation loss is also high
- model is not learning enough structure

Possible reasons:
- model too simple
- training too short
- learning rate too low
- features not informative enough

#### Good Fit
A healthier training pattern looks like:
- training loss goes down
- validation loss also goes down
- gap between them is not too large

That suggests the model is learning useful patterns that generalize.

#### Overfitting
Overfitting means the model is fitting training data too specifically.

Signs:
- training loss keeps going down
- validation loss stops improving or starts going up
- gap between training and validation grows

That suggests the model is memorizing rather than generalizing.

---

### ***14) A Tiny Curve Intuition***

#### Healthy pattern

```text
epoch 1: train loss 0.90   val loss 0.95
epoch 2: train loss 0.75   val loss 0.80
epoch 3: train loss 0.60   val loss 0.67
epoch 4: train loss 0.50   val loss 0.58
```

This looks healthy:
- both are going down
- validation is improving too

#### Overfitting pattern

```text
epoch 1: train loss 0.90   val loss 0.95
epoch 2: train loss 0.70   val loss 0.78
epoch 3: train loss 0.50   val loss 0.76
epoch 4: train loss 0.30   val loss 0.89
```

This looks risky:
- training keeps improving
- validation starts getting worse

That is the classic early overfitting signal.

---

### ***15) Why Learning Can Become Unstable***

Sometimes training does not improve smoothly.

Possible reasons:

#### Learning rate too high
If updates are too large:
- loss may jump around
- training may overshoot
- optimization may diverge

#### Bad initialization or fragile setup
Sometimes parameter starting points or model choices make optimization harder.

#### Data issues
Noisy labels, poor scaling, or weak features can make training behavior messy.

#### Batch behavior
Small batches can make training noisier.
That is not always bad, but it can make curves wobblier.

So if training looks chaotic, it does not always mean “the model is broken.”
It often means the optimization setup needs adjustment.

### ***16) A Minimal PyTorch-Style Example***

```python
for epoch in range(num_epochs):
    model.train()

    for X_batch, y_batch in train_loader:
        optimizer.zero_grad()

        y_pred = model(X_batch)
        loss = loss_fn(y_pred, y_batch)

        loss.backward()
        optimizer.step()
```

This is the minimal training loop.

If we want validation too, we usually add a separate evaluation phase after the training loop for that epoch.

Example shape:

```python
for epoch in range(num_epochs):
    model.train()
    # training loop

    model.eval()
    # validation loop
```

So:
- `train()` mode for training behavior
- `eval()` mode for evaluation behavior

Day 7 will make more practical use of this.

### ***17) Why `train()` and `eval()` Matter***

Some layers behave differently during training and evaluation.

Examples:
- dropout
- batch normalization

So in frameworks like PyTorch:
- `model.train()` tells the network to behave in training mode
- `model.eval()` tells it to behave in evaluation mode

Even before going deep into those layer types, we should know this distinction exists.

It is part of writing a correct training workflow.

### ***Simple Summary***

A neural network learns through a training loop.

For each batch:
- clear old gradients
- run a forward pass
- compute loss
- run backpropagation
- update parameters

After all batches are processed once, that is one epoch.

Training loss tells us how well the model is fitting training data.
Validation loss tells us how well it generalizes to held-out data.

If both improve, learning is usually healthy.
If training improves but validation worsens, overfitting may be starting.
If neither improves much, the model may be underfitting or training poorly.

So the basic loop is:

```text
batch → forward pass → loss → backward pass → optimizer step
```

That is the operational heart of neural network training.
