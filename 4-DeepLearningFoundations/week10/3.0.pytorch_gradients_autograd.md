# 2. 🧠 Why Gradients Matter

In Week 9, we learned the basic training idea:

```text
prediction → loss → gradient → parameter update
```

A neural network has parameters:

```text
weights
biases
```

The model makes predictions using those parameters.

Then we compute a loss.

The loss tells us how wrong the predictions are.

But the loss alone does not tell us how to change each weight.

For training, we need to know:

```text
If this weight changes slightly, how does the loss change?
```

That sensitivity is the gradient.

A gradient answers:

```text
Which direction should this parameter move to reduce the loss?
```

So the training process needs gradients for every trainable parameter.

Without gradients, the optimizer would not know how to update weights.

# 3. ⚙️ What Autograd Means

Autograd means automatic differentiation.

In simple terms:

> PyTorch can automatically track tensor operations and compute gradients for us.

Without autograd, we would have to manually derive gradients for every operation.

For a tiny example, that may be possible.

For a neural network with many layers, activations, losses, and thousands or millions of parameters, manual gradient calculation would be impractical.

PyTorch handles this by building a computation graph.

That graph remembers how a result was produced.

Then, when we call:

```python
loss.backward()
```

PyTorch walks backward through that graph and computes gradients.

This is the implementation version of the backpropagation idea from Week 9.

<br>

# 4. 🏷️ `requires_grad=True`

By default, PyTorch does not track gradients for every tensor.

That would waste memory and computation.

We only need gradients for tensors that should be learned or optimized.

To ask PyTorch to track gradients for a tensor, we use:

```python
requires_grad=True
```

Example:

```python
import torch

x = torch.tensor(2.0, requires_grad=True)

print(x)
print(x.requires_grad)
```

Expected output:

```text
tensor(2., requires_grad=True)
True
```

This tells PyTorch:

```text
Track operations involving x because we may later need the gradient of some result with respect to x.
```

# 5. 🧮 A Simple Gradient Example

Let’s start with a simple function:

```text
y = x²
```

In PyTorch:

```python
import torch

x = torch.tensor(3.0, requires_grad=True)
y = x ** 2

print("x:", x)
print("y:", y)
```

Here:

```text
x = 3
y = 3² = 9
```

Now we ask PyTorch to compute the gradient:

```python
y.backward()
```

Then we inspect:

```python
print(x.grad)
```

Expected output:

```text
tensor(6.)
```

Why is the gradient 6?

Because:

```text
y = x²
```

The derivative is:

```text
dy/dx = 2x
```

At:

```text
x = 3
```

we get:

```text
2 × 3 = 6
```

So PyTorch correctly computes:

```text
x.grad = 6
```

# 6. 🔗 Computation Graph Intuition

When we write:

```python
x = torch.tensor(3.0, requires_grad=True)
y = x ** 2
```

PyTorch remembers that `y` came from `x` through a square operation.

Conceptually:

```text
x → square → y
```

This remembered chain is the computation graph.

When we call:

```python
y.backward()
```

PyTorch walks backward:

```text
y → square → x
```

and calculates:

```text
How much does y change with respect to x?
```

That result is stored in:

```python
x.grad
```

The key idea:

> Forward operations build the graph. `.backward()` walks backward through the graph and computes gradients.

# 7. 🧩 A Slightly Longer Example

Now let’s use multiple operations.

```python
import torch

x = torch.tensor(2.0, requires_grad=True)

y = x * 3
z = y + 4
loss = z ** 2

print("x:", x)
print("y:", y)
print("z:", z)
print("loss:", loss)
```

Let’s calculate the forward pass manually.

```text
x = 2
```

Then:

```text
y = x × 3 = 2 × 3 = 6
```

Then:

```text
z = y + 4 = 6 + 4 = 10
```

Then:

```text
loss = z² = 10² = 100
```

So forward pass result is:

```text
loss = 100
```

Now we call:

```python
loss.backward()
print(x.grad)
```

Expected gradient:

```text
tensor(60.)
```

Why?

We can reason with the chain rule:

```text
loss = z²
z = y + 4
y = 3x
```

So:

```text
dloss/dz = 2z = 20
dz/dy = 1
dy/dx = 3
```

Chain rule:

```text
dloss/dx = dloss/dz × dz/dy × dy/dx
```

So:

```text
dloss/dx = 20 × 1 × 3 = 60
```

PyTorch computes this automatically.

This is the same backpropagation idea from Week 9.

# 8. 📦 Where Gradients Are Stored

After we call `.backward()`, gradients are stored in the `.grad` attribute of tensors that require gradients.

Example:

```python
x = torch.tensor(3.0, requires_grad=True)
y = x ** 2

y.backward()

print(x.grad)
```

Output:

```text
tensor(6.)
```

Important:

```text
x.grad stores the gradient of y with respect to x.
```

More generally, if we call:

```python
loss.backward()
```

then:

```text
parameter.grad stores the gradient of loss with respect to that parameter.
```

That is exactly what optimizers use.

# 9. ⚠️ Gradients Accumulate by Default

This is one of the most important PyTorch behaviors.

Gradients accumulate unless we clear them.

Example:

```python
import torch

x = torch.tensor(3.0, requires_grad=True)

y = x ** 2
y.backward()
print("After first backward:", x.grad)

z = x ** 2
z.backward()
print("After second backward:", x.grad)
```

Output:

```text
After first backward: tensor(6.)
After second backward: tensor(12.)
```

Why did it become 12?

The gradient of `x²` at `x = 3` is 6.

But PyTorch added the second gradient to the first gradient.

```text
6 + 6 = 12
```

So gradients accumulate.

This is not a bug.

It is intentional behavior.

But in normal training loops, we usually do not want gradients from previous batches to mix with gradients from the current batch.

That is why we clear gradients before each new backward pass.

<br>

# 10. 🧹 Clearing Gradients Manually

For a single tensor, we can clear the gradient like this:

```python
x.grad = None
```

Example:

```python
x = torch.tensor(3.0, requires_grad=True)

y = x ** 2
y.backward()
print("First gradient:", x.grad)

x.grad = None

z = x ** 2
z.backward()
print("Second gradient after clearing:", x.grad)
```

Output:

```text
First gradient: tensor(6.)
Second gradient after clearing: tensor(6.)
```

Now the second gradient does not accumulate on top of the first one.

# 11. 🔄 Why Training Loops Use `zero_grad()`

In real model training, we usually do not manually set each parameter’s `.grad` to `None`.

Instead, we use:

```python
optimizer.zero_grad()
```

A typical training step looks like this:

```python
optimizer.zero_grad()

predictions = model(X_batch)
loss = loss_fn(predictions, y_batch)

loss.backward()
optimizer.step()
```

The order matters.

```text
1. Clear old gradients
2. Run forward pass
3. Compute loss
4. Run backward pass
5. Update parameters
```

The first step prevents old gradients from mixing with new gradients.

This directly connects to the Week 9 training-loop pattern:

```text
zero_grad → forward → loss → backward → step
```

# 12. 🚫 `torch.no_grad()`

Sometimes we do not want PyTorch to track gradients.

For example:

```text
validation
testing
inference
simple inspection
```

During these phases, we are not updating model weights.

So gradient tracking is unnecessary.

We can disable it with:

```python
with torch.no_grad():
    predictions = model(X_val)
```

This saves memory and computation.

It also makes our intent clear:

```text
We are only evaluating or predicting, not training.
```

Important distinction:

```text
model.eval() changes model behavior for certain layers.
torch.no_grad() disables gradient tracking.
```

They are related during evaluation, but they are not the same thing.

We usually use both during validation or testing:

```python
model.eval()

with torch.no_grad():
    predictions = model(X_val)
```

# 13. 🧠 Autograd and Neural Networks

In a neural network, we do not usually create one tensor like `x` and optimize it directly.

Instead, the model has parameters:

```text
layer weights
layer biases
```

Those parameters have:

```text
requires_grad=True
```

When we compute loss and call:

```python
loss.backward()
```

PyTorch computes gradients for all trainable model parameters.

Then the optimizer uses those gradients:

```python
optimizer.step()
```

So the full idea is:

```text
forward pass builds the computation graph
loss measures error
backward computes gradients
optimizer updates parameters
```

This is PyTorch’s practical implementation of backpropagation.

# 16. ✅ Answer Key

## Answer 1

`requires_grad=True` tells PyTorch:

```text
Track operations involving this tensor because we may need gradients later.
```

This allows PyTorch to include the tensor in the computation graph.

<br>

## Answer 2

`x.grad` should be:

```text
tensor(6.)
```

Reason:

```text
y = x²
dy/dx = 2x
x = 3
2 × 3 = 6
```

So the gradient is 6.

<br>

## Answer 3

`.backward()` walks backward through the computation graph and computes gradients.

In training, when we call:

```python
loss.backward()
```

PyTorch computes gradients of the loss with respect to trainable tensors or model parameters.

<br>

## Answer 4

PyTorch stores gradients in the `.grad` attribute.

Example:

```python
x.grad
```

For model parameters, each parameter has its own `.grad` after backpropagation.

<br>

## Answer 5

Training loops use `optimizer.zero_grad()` because gradients accumulate by default.

If we do not clear old gradients, gradients from previous batches mix with gradients from the current batch.

The usual pattern is:

```text
zero_grad → forward → loss → backward → step
```

<br>

## Answer 6

`torch.no_grad()` disables gradient tracking.

We use it during:

```text
validation
testing
inference
```

because we are not training or updating weights during those phases.

It saves memory and computation.
